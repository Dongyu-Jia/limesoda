# GitHub Issue Backlog: Milestones 1 & 2

This document provides the granular execution plan for the Limesoda "High-Speed" roadmap.

---

## 🟢 Epic 1: Milestone 1 - Foundation & Quality

### 1. [M1-Core] Initialize Postgres Schema & Prisma Client
**Description**:
Implement the baseline relational schema in `schema/prisma/schema.prisma` as defined in the Architecture RFC.
- **Entities**: `Organization`, `Project`, `TeamMember`, `Secret`.
- Ensure all foreign keys, unique constraints (especially `githubRepoUrl`), and default timestamps are correctly defined.
- Run `prisma generate` and create the initial migration file.

**Acceptance Criteria**:
- `schema.prisma` is valid and formatted.
- Initial migration file is generated and correctly reflects the relational model.
- Prisma Client can be successfully imported in a test Python script.

### 2. [M1-Core] Establish GitHub Actions CI/CD (Linter + Tests)
**Description**:
Create the foundational CI pipeline using GitHub Actions to ensure code quality from day one.
- **Workflow**: `.github/workflows/ci.yml`
- **Jobs**:
    - `lint`: Run `flake8` / `black` for Python and `eslint` / `prettier` for React.
    - `test-api`: Run `pytest` for the FastAPI backend.
    - `test-web`: Run `vitest` for the React frontend.
- Ensure the pipeline triggers on every Pull Request and move to the `main` branch.

**Acceptance Criteria**:
- CI pipeline triggers on a test commit.
- All lint and test jobs report success/failure correctly in the GitHub interface.

## 🔵 Epic 2: Milestone 2 - Agentic Loop (Walking Skeleton)

### 3. [M2-Engine] Temporal Workflow & State Machine
**Description**:
Define the core `LimesodaWorkflow` in Python using the Temporal SDK.
- Implement the state machine logic: Triage Phase -> Coding Phase -> Human Review.
- Support Temporal Signals (e.g., `issue_assigned`) to trigger or resume the workflow.
- Create activity stubs for sandbox management and agent execution.

**Acceptance Criteria**:
- A "Hello World" workflow can be successfully started and completed via the `temporal` CLI or Web UI.
- The workflow correctly transition between mock states.

### 4. [M2-Engine] E2B Sandbox Lifecycle Manager (Activity)
**Description**:
Implement the Temporal activity that manages the lifecycle of agent sandboxes.
- Use the **E2B Python SDK** to create and destroy sandboxes.
- Action: Create a sandbox instance, execute a simple shell command (e.g., `git --version`), and return the output.
- Ensure proper cleanup (sandbox destruction) even if the activity fails.

**Acceptance Criteria**:
- Temporal activity can successfully boot an E2B sandbox.
- Activity returns the correct shell stdout.
- Sandbox is confirmed closed after execution.

### 5. [M2-Brain] Manager Agent (Triage Node)
**Description**:
Implement the `Manager` agent as a LangGraph node.
- Input: Takes a GitHub Issue description (text).
- Logic: Analyzes the requirement and categorizes it (e.g., "Refactor", "Bugfix").
- Output: Returns a structured JSON classification and signals the start of the implementation node.

**Acceptance Criteria**:
- The node takes text input and returns a dictionary with `category` and `rationale`.
- Integrated as a step in the overall LangGraph.

### 6. [M2-Brain] Coding Agent (Implementation Node)
**Description**:
Implement the `Coder` agent as the primary implementation LangGraph node.
- Logic: Takes the categorized issue, boots an E2B sandbox, and executes the `claude` CLI tool inside the sandbox to perform a code edit.
- Output: Returns the file `diff` generated by the agent.

**Acceptance Criteria**:
- Agent can modify a file inside the E2B sandbox.
- The workflow receives the correct diff output.

### 7. [M2-Ingress] GitHub Webhook Handler (Reactive Trigger)
**Description**:
Implement the `POST /api/webhooks/github` endpoint to react to human signals on GitHub.
- Auth: Implement HMAC verification (`X-Hub-Signature-256`).
- Action: Listen for the `issues:assigned` payload.
- Logic: If an issue is assigned to the Limesoda bot, identify the Project and start/signal the corresponding Temporal workflow.

**Acceptance Criteria**:
- Endpoint returns `200 OK` for valid signatures.
- A mock payload successfully triggers a new Temporal workflow execution.
